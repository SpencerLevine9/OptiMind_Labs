# OptiMind Labs

OptiMind Labs is a reinforcement learning project for intelligent HVAC control in a simulated commercial building environment. The system trains a PPO-based agent to make HVAC control decisions using environmental observations such as indoor temperature, outdoor temperature, time of day, occupancy, and price signal.

The goal of the project is to study whether a learned controller can balance:

- occupant comfort
- energy cost
- long-term control performance

The project also includes comparison against a simple rule-based baseline, trajectory logging, preprocessing, evaluation, and figure generation for analysis and presentation.

Environment Description

The custom HVAC environment is implemented in:

src/envs/hvac_env.py

The environment state contains 5 features:

indoor temperature
outdoor temperature
time of day
occupancy
price signal

The action space is discrete and represents HVAC control decisions.

The reward is designed to balance thermal comfort and energy-related cost.

How to Run the Full Pipeline

Run all scripts from the project root using module mode.

Use this form:

python -m scripts.<script_name>

Do not rely on:

python scripts/<script_name>.py

Some scripts use project-root imports such as from src.envs.hvac_env import HVACEnv, so module execution is the reliable way to run the pipeline.

1. Train the PPO model
python -m scripts.train

Expected output:

models/ppo_hvac.zip

This step trains the PPO agent through interaction with the custom HVAC environment.

2. Collect PPO trajectories
python -m scripts.collect_data

Expected output:

data/trajectories.csv

This file contains rollout data generated by the trained PPO policy.

3. Collect random-policy trajectories
python -m scripts.collect_random_data

Expected output:

data/random_trajectories.csv

This file contains rollout data generated by a random policy and is used for comparison and preprocessing.

4. Preprocess trajectory data
python -m scripts.preprocess

Expected output:

data/random_trajectories_processed.csv

Optional behavior:

if a PPO-specific processed input path is available in the script, an additional PPO processed file may also be written

By default, preprocessing applies z-score normalization to state and next-state features. The script also supports optional reward clipping/scaling behavior.

5. Generate dataset profile
python -m scripts.data_profile

Expected output:

results/dataset_profile.csv

This step compares the random-policy and PPO trajectory datasets and reports:

schema consistency
missing values
outlier counts
descriptive statistics
6. Evaluate PPO vs baseline
python -m scripts.evaluate

Expected outputs:

results/final_metrics.csv
results/final_metrics.md

This step compares:

PPO controller
rule-based baseline controller

Metrics include:

average total reward
standard deviation of total reward
average energy cost
standard deviation of energy cost
comfort violation rate
number of evaluation episodes
7. Generate figures and EDA outputs
python -m scripts.generate_figures

Expected outputs in figures/ include items such as:

histograms
boxplots
reward distribution
episode length distribution
time-series plots
correlation heatmap
PCA/state-space plots
eda_insights.txt

This step creates the visual assets used for analysis and presentation.

What Each Main Script Does
train.py
Trains the PPO model using the HVAC environment.
collect_data.py
Runs the trained PPO policy and records rollout trajectories.
collect_random_data.py
Runs a random policy and records rollout trajectories.
preprocess.py
Normalizes and prepares trajectory data for analysis/downstream use.
data_profile.py
Compares dataset structure and descriptive statistics between PPO and random rollouts.
baseline_controller.py
Implements the rule-based baseline controller.
evaluate.py
Compares PPO and baseline performance using final summary metrics.
generate_figures.py
Produces plots and EDA summary outputs for the report/slides.
Notes
This is a reinforcement learning project, so the PPO model is trained from interaction with the environment rather than from a fixed pre-collected dataset.
The trajectory CSV files are generated after or during policy execution for logging, analysis, comparison, and evaluation.
If an import error occurs when running a script directly, rerun it in module form from the project root.
